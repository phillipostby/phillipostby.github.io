Disabling that Pesky Display Stream Compression (DSC)

So, just to set the scene, you just purchased that brand new 6k monitor, but your graphics card just is not quite powerful enough to drive a modern DX12 or Vulkan program on it. That is fine, modern cards allow you to do something called Integer Scaling on the GPU side.

But, what is this? Regardless of what vendor you have (Nvidia, AMD, or Intel,) all of the integer scaling options are completely missing. Most likely, it is because of Display Stream Compression which appears to affect all vendors equally. You did not even want to run it at 144hz, you just wanted to use it at 1080p at 60hz, but whoops! Looks like you will not be able to.

The good news is that it appears you can set keys to disable DSC to get around this under the display adapter itself:
[HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Class{4d36e968-e325-11ce-bfc1-08002be10318}\XXXX]
XXXX appearing to refer to the current GPU/adapter in question. With Intel, you can add the following key and it will disable DSC and potentially MST depending on if you set it to 1 or 2:
[HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Class{4d36e968-e325-11ce-bfc1-08002be10318}\XXXX] "DPMstDscDisable"=dword:00000001
With AMD, you can set the following value instead:
[HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Class{4d36e968-e325-11ce-bfc1-08002be10318}\XXXX] "DalDisableDSC"=dword:00000001
If I find out what Nvidia's value is, I will update it here as well. However, as of presently, it does not look like one publicly exists.

AI Analysis of Bytes Using ChatGPT

After using the tool I mentioned last time, I came across a GitHub portfolio of someone I was researching. For the most part, the portfolio had been completely abandoned to time, but my assumption is that the user was using it to store examples for their own cybersecurity related research:


The biggest mystery was a repository holding a series of files without any context at all called "shellcode.txt," which seems to suggest that they are related to using the shell (which could honestly be anything, Bash, Powershell, CMD, so forth.) Another repository suggests that this user was working with a mainly client side driven administration tool for a website/browser, so perhaps the two are connected?


In the past, I hosted various web servers (usually based on Apache and NGINX,) and would see various injection attempts made via HTTP requests. Researching them usually lead to the conclusion that they were actually x86-64 machine code exploits, and knowing this particular individual, seemed likely. Each file was exactly as shown, a byte array was created, and was then appended to line by line. My first guess is that there might be a string in here (encoded in ASCII or a version of UTF) which might be human readable:


Other sections yielded similar results, if this was text, it was encrypted - but most likely it was something else entirely. Another idea is that it could be an image, either a uncompressed bitmap, or maybe a JPEG (which you can tell by looking at the first three bytes of a file, FF D8 FF.)


After some discourse with people I had contact with, 7thdrxn actually had the idea to use ChatGPT to try parsing and identifying the first file. Many of the ML learning models seem (presently) overhyped, but ChatGPT has been trained using a significant number of Stack Overflow posts (which if these files are indeed some sort of exploit, there are likely some posts discussing this.) Sure enough, not only did it recognize what the first file was, but actually attempted to provide a breakdown of how it operated:
This appears to be a PowerShell script that initializes an array of bytes called $buf with a long sequence of hexadecimal values.

Without knowing the context in which this code is being used, it's difficult to say exactly what it does. However, the fact that it contains opcodes for x64 assembly suggests that it is likely intended to be injected into a process as shellcode.

Some of the notable instructions in the shellcode include:

0xfc,0x48,0x83,0xe4,0xf0,0xe8,0xcc,0x0,0x0,0x0: These instructions set up the stack frame for the shellcode by allocating space on the stack and storing the previous value of the stack pointer.

0x48,0x31,0xd2: This instruction clears the RDX register.

0x48,0xff,0xc9: This instruction decrements RCX by 1.

0xff,0xe0: This instruction calls the function whose address is stored in the EAX register.

Without further information, it is difficult to provide a more detailed explanation of what this shellcode does.

Line 1 initializes a byte array called $buf that will hold the shellcode.

Lines 2-11 represent the first part of the shellcode, which is a function prologue that sets up the stack and prepares for the function to be called.

Lines 12-23 represent the second part of the shellcode, which extracts the function pointers from the PEB (Process Environment Block) and sets them up to be called later.

Lines 24-43 represent the third part of the shellcode, which sets up a loop that will iterate through the function pointers and call each one in turn.

Line 24 sets the starting address of the loop to the address of the first function pointer.

Line 25 extracts the current function pointer and saves it in the edx register.

Line 26 extracts the second function pointer and saves it in the ebx register.

Line 27 extracts the third function pointer and saves it in the ecx register.

Line 28 calls the current function pointer by using the call instruction.

Line 29-31 check whether the current function call was successful by checking the return value in the eax register. If the return value is greater than 1, the loop is exited.

Line 32-33 add the size of a function pointer to the current function pointer, effectively moving the loop to the next function pointer.

Line 34-38 are a special case that checks whether the loop has reached the end of the function pointer array. If the end of the array has been reached, the loop is exited.

Line 39-43 jump back to the start of the loop to continue calling the remaining functions in the array.

Lines 44-50 are an epilogue that cleans up the stack and returns control to the calling function.
Sure enough, this suggests that this is x86-64 machine code which is designed to make note of function calls, and call these in some sort of loop. This in itself is not necessarily malicious, but obviously is some form of injection attempt that may be combined with some of the other files in the repository. As a note though, ChatGPT's analysis above may or may not be completely accurate, and as with any AI driven tool, it is up to the researcher to confirm manually what it assessed. However, it is clear how it can be used as a powerful analysis tool and save time. Finally, I decided to just do literal searches for the content in Google directly:


Obviously this is what I should have done from the start. The pages yielded all confirm that this is indeed a PowerShell exploit which is to be combined with other payloads/files. Different versions of it can be used to launch a type of RAT/Remote Access Tool on the victim's machine, which you can use some of the exploits from Metasploit with.

WhatsMyName OSINT Tool

Lots of people have used/contributed to this tool in the past, but surprisingly I had not heard of it until reading through Mike Meyer's Security+ SY0-601 study guide. One way to gather open source intelligence is to comb through instances of a particular username. You can manually check it, or punch it into a search engine like Google (provided it is even present in the search results,) but the WhatsMyName tool makes this exceptionally easy:


In this example, I'm going to generically search for someone called "Testman," to which there are likely to be plenty of results for. However, before we do that, you can click the filter and adjust it to various categories - porn is generally excluded by default:

Lenovo BIOS Simulator

Recently I was looking into purchasing a Lenovo P15v Gen 3 due to it being one of the very few laptops with SODIMM memory (possibly letting me install 48GB or even 64GB modules as they come out for DDR5,) and NVMe raid for dual Optane P1600x SSDs. However, I still use Windows 10, and would prefer not to mess with asymmetric thread scheduling due to the hybrid Intel Alder Lake processor.

In theory, you can simply set the number of CPUs to 2 x N, where N is the number of performance cores, and 2 indicates SMP/hyperthreading under boot.ini - any cores after this should be the efficient cores, and as a result, Windows will never be aware of the efficient cores at all. However, I did not really trust this option, and do not have enough familiarity with the core ordering numbers in Alder Lake CPUs.

As an alternative, BIOSes often will let you disable cores, and many Intel boards will also let you specifically disable the efficient cores. The issue with this though, it is often very vendor implementation specific, and it's a complete grab bag as to whether the UEFI/BIOS setup utility will even let you do this.

After digging around though, it turns out Lenovo has a UEFI/BIOS simulator that lets you actually preview what settings are available:


Digging through the options, I was able to find the Intel (Alder Lake) version of the Lenovo P15v Gen3, and started looking around at the menus. Turns out, no, according to the utility, it will not let you turn off the efficient cores. This discovery possibly saved me thousands:

Ultimate Performance Plan Missing

One of the first things I do when installing Windows is to ensure that I am using the Ultimate Performance power plan - this generally reduces latency/switching time of activating turbo boost like features, among disabling other power saving mechanisms.


Most of the time, you can activate it in the Power Options control panel applet by using the following:
powercfg -duplicatescheme e9a42b02-d5df-448d-aa00-03f14749eb61
However, it might not show up depending on how the computer is configured, you might wind up only seeing Balanced only with this as an example.

In addition, trying to force it does not really work either, using the following to force the plan as the current one only results in a "Not Supported" message, even after installing platform specific chipset and ACPI drivers:
powercfg -s e9a42b02-d5df-448d-aa00-03f14749eb61
After further research, it turns out that more modern platforms have a type of ultra low power standby activated by default called S0 Standby (referring to the sleep state,) allowing you to put the computer to lower power sleep - but the trade off is tighter Windows/UEFI integration. After changing this value from "Windows 10" to "Linux" on my Lenovo L14 Gen 2 AMD, I can now set and use the Ultimate Performance power plan just like I did in the past.
